Chapter 17

KV Cache

Why generating text is wastefully slow without caching — and the elegant trick that fixes it by trading memory for speed.

How Many Operations Does It Take?

Let's count the floating-point operations (FLOPs) needed to generate a response to "Please tell me about Paris." The prompt is 5 tokens; the completion might be 20 tokens. For our 20-layer model, each next-token prediction requires roughly 5 billion FLOPs just for the matrix multiplications.

The Waste Problem

Here's the issue with the generation loop from Chapter 9. To generate token 7, we feed in all 7 tokens (prompt + 2 generated). The model processes all 7 tokens through all 20 layers. But we only use the prediction from the last position! The first 6 positions were already computed when generating tokens 5 and 6 — that work is completely wasted.

FLOP Count Without KV Cache (20-token completion)
1
Token 6: process 6 tokens → ~5.0B FLOPs
2
Token 7: process 7 tokens → ~5.7B FLOPs
3
Token 8: process 8 tokens → ~6.3B FLOPs
...
Each step recomputes everything from scratch
Σ
Total: ~277 billion FLOPs for 20 tokens

The KV Cache Solution

The insight: inside the attention mechanism, each position computes Key and Value vectors. These don't change when we add a new token to the end — position 3's Key and Value are the same regardless of whether the sequence has 5 tokens or 50 tokens (thanks to causal masking, position 3 can't see anything after itself).

The KV Cache Trick

Store the K and V vectors from all previous tokens. When generating the next token, only compute Q, K, V for the new token. Use the cached K and V from all previous positions for the attention calculation. This means we process one token through the network instead of the entire sequence.

FLOP Count WITH KV Cache
1
First pass (prompt): process 5 tokens normally → ~5.0B FLOPs (cache K,V)
2
Token 6: process 1 token only → ~1.0B FLOPs
3
Token 7: process 1 token only → ~1.0B FLOPs
...
Each subsequent token: ~1B FLOPs (not growing!)
Σ
Total: ~5B + 20 × 1B = ~25 billion FLOPs

That's roughly 11× fewer FLOPs for a 20-token completion. For longer completions (100+ tokens), the savings are even more dramatic — potentially 50× or more.

The Trade-Off: Memory

The KV cache saves compute but costs memory. For each token in the sequence, across all layers and all heads, we need to store the K and V vectors. For our 20-layer model with D=1280:

KV Cache Memory Per Token
1
K and V vectors: 2 × D = 2 × 1280 = 2560 numbers per layer
2
Across 20 layers: 2560 × 20 = 51,200 numbers per token
3
At 2 bytes each (BF16): 51,200 × 2 = ~100 KB per token
!
For a 2048-token sequence: ~200 MB. For 8 parallel users: ~1.6 GB

This memory trade-off is manageable — GPUs have 40–80 GB of memory. The compute savings are well worth it, especially when serving millions of users.

Key Insight

The KV Cache trades memory for speed. By storing the Key and Value vectors from all previously processed tokens, each new token only needs to be processed once through the network — instead of reprocessing the entire growing sequence. This makes text generation dramatically faster (10–100×) and is essential for serving AI models to millions of users in real-time.
