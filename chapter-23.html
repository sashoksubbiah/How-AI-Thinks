Chapter 23

Multiple GPUs

When one GPU isn't enough — how to split training across 8, 64, or thousands of chips working in parallel.

Our 560-million parameter model, plus optimizer state (Adam stores 2 extra numbers per parameter), plus the activations saved for backprop, plus the gradient — all need to fit in GPU memory. A single NVIDIA H100 has 80 GB. For our model, one GPU is enough. But for larger models (GPT-3: 175 billion parameters), you need many GPUs working together.

Three Parallelism Strategies

📦

Data Parallelism

Copy the entire model to each GPU. Each GPU processes a different batch of data. After each step, GPUs synchronize their gradients (average them) and apply the same weight update. Simple and effective.

✂️

Model Parallelism

Split the model's layers across GPUs. GPU 1 runs layers 1–10, GPU 2 runs layers 11–20. Data flows between GPUs. Needed when the model is too big for one GPU's memory.

🔗

FSDP (Fully Sharded)

Shard both model weights and optimizer state across GPUs. Each GPU stores only 1/Nth of the parameters. They share pieces as needed during forward/backward passes. Maximizes memory efficiency.

Practical Setup

For our 32-layer model, we'll use 8 NVIDIA H100 GPUs in a single node, connected by high-speed NVLink (900 GB/s between GPUs). Using FSDP, each GPU stores 1/8th of the weights and optimizer state. Cost: roughly $2–3 per GPU per hour, so 8 GPUs for a multi-day training run costs $500–2,000.

The largest models (GPT-4, Claude) use thousands of GPUs across multiple data centres, with sophisticated combinations of all three parallelism strategies plus pipeline parallelism (different batches at different stages simultaneously). Training costs run into tens of millions of dollars.

Key Insight

When one GPU isn't enough, training splits across multiple GPUs using data parallelism (same model, different data), model parallelism (different layers on different GPUs), and FSDP (shard everything). Our 32-layer model uses 8 H100 GPUs. The biggest models use thousands.
